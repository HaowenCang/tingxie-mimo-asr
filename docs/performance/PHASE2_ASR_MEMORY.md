# 阶段 2：ASR 内存与主进程吞吐优化结果

> 分支：`codex/perf-asr-memory`
>
> 基线：`main` / `de74283`
>
> 日期：2026-07-20

## 完成内容

- 音频切片在 RPM/全局冷却许可之前只保留文件路径和字节数，不读取文件、不创建 Base64、不创建 JSON 请求体。
- 每次请求与重试均按“RPM 许可 → 字节额度 → 读取与编码 → 发送 → 释放额度”执行。
- 新增全局 FIFO 字节级内存闸门，按系统空闲内存计算 128–192 MiB 预算。
- 取消等待中的任务会从内存队列移除，不消耗额度，也不阻塞后续任务。
- Base64 10 MB 校验改为使用文件大小的精确编码长度公式，无需提前读取文件。
- 429/RPM 等待仍由既有独立计数处理，不消耗真实错误重试次数；等待期间不持有 Base64/JSON。
- 诊断日志改为最多每 150ms 或累计 32 条批量追加；日志大小只在首次写入时读取，后续使用缓存计数。
- 应用退出前强制刷新日志队列，轮转仍保留 5 MiB × 3 份且继续脱敏。

## 内存边界

以 6.4 MiB 目标切片和 30 个已进入调度池的任务估算：

| 场景 | 旧实现 | 新实现 |
|---|---:|---:|
| 等待 RPM 时的 Base64 | 最多约 256 MiB | 0 |
| 编码/请求大对象 | 随任务并发增长，未设字节上限 | 128–192 MiB 硬预算 |
| 6.4 MiB 切片估算预留 | 无 | 约 25.66 MiB/请求 |
| 同时编码/发送数量 | 由任务并发决定 | 约 4–7 个，按字节而非任务数 |

30 任务确定性测试已验证：100 字节预算、每任务预留 25 字节时，峰值并行准备/发送严格为 4，结束后额度全部归还。

## 流式 Base64 结论

本阶段不启用流式 JSON/Base64 请求体。当前 Electron/Node `fetch` 需要额外处理半双工请求、`Content-Length`、代理兼容和失败重放；在未对小米按量与 Token Plan 进行真实计费兼容性验证前，收益不足以抵消发布风险。采用已验证的“RPM 后编码 + 字节预算”方案作为稳定落点。

## 正确性验证

- 限流许可前不执行 payload 准备。
- 内存不足时 FIFO 等待；取消后安全移除。
- 所有成功、网络失败、503、429 和质量失败路径均通过 `finally` 释放额度。
- 96/96 自动化测试通过，共 25 个测试文件。
- TypeScript、Vite 和 Electron 生产构建通过。
- `npm audit --omit=dev`：0 个生产依赖漏洞。
