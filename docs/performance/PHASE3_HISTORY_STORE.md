## 完成内容

- 将单一 `history.json` 迁移为 `history/index.json` 与 `history/records/<id>.json`。
- Renderer 启动只接收 `TranscriptSummary[]`；完整正文在用户打开记录时按 ID 加载。
- 媒体库、历史列表和状态筛选改为只消费摘要，不再常驻全部正文、段落和切片。
- 单段正文编辑通过 `transcriptId + segmentId + patch` 保存，只重写目标记录和轻量索引。
- 新建转写、AI 分析、媒体重新关联、删除、播放 URL 和质量修复统一改用分离式存储。
- Main 不再缓存完整历史数组，只缓存轻量索引；当前详情仍由 Renderer 按需持有。

## 迁移与数据安全

1. 读取旧 `history.json`，解析失败时立即停止，不创建空索引。
2. 校验数组、关键字段、唯一 ID、记录数与总文本长度。
3. 创建校验备份：`backups/history-before-split-store-0.12.json`。
4. 每条记录通过临时文件原子写入，并立即回读校验。
5. 全部记录通过后才原子写入 `history/index.json`，该索引是新存储的提交点。
6. 迁移被中断时，下次启动可重复执行；已有临时/记录文件不会使旧数据失效。
7. 磁盘或备份写入失败时，不提交索引，并以内存旧格式继续显示历史。
8. 旧 `history.json` 至少保留一个正式版本周期，不删除、不覆盖；迁移备份本身可直接供旧安装包恢复。

质量修复的备份由整库副本调整为目标记录独立备份，避免每次维护复制整个历史文件。

## 回退方法

本阶段通过独立 PR 合并。若新存储出现重大问题，可整体回退 PR 并将 `backups/history-before-split-store-0.12.json` 复制回 `history.json`。新目录不会覆盖旧文件；若回退前已在新版本新增或编辑记录，应先从 `history/records` 导出最新兼容快照，再安装旧版本。
