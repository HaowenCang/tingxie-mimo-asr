## 1. 目标与边界

本方案用于系统性优化 Windows Electron 应用在以下场景中的性能：

- 超长转写结果的打开、滚动、编辑和音频同步播放。
- 大量历史记录和媒体文件的启动加载、搜索与管理。
- 高并发 ASR 请求期间的内存占用与主进程响应能力。
- 长 AI 对话及流式 Markdown 输出的渲染效率。
- Liquid Glass 视觉效果带来的 GPU 合成和滚动成本。
- 前端启动包体积、字体资源和按需加载。

本方案不改变 MiMo ASR 的识别语义、切片质量策略、RPM 限流原则、失败切片容错规则或用户数据内容。

开始实施前必须先完成 GitHub 安全基线，禁止直接在当前脏工作区继续叠加大规模修改。

## 3. 已确认的性能问题

## 3.1 P0：播放音频时重复渲染完整转写详情页

当前播放器的 `timeupdate` 同时更新播放器本地状态和 `TranscriptDetail` 顶层 `currentTime`。

每次更新时间会：

1. 线性扫描全部段落寻找当前播放段。
2. 重新执行整个转写详情组件。
3. 重新协调最多 1,191 个时间轴节点和文本框。
4. 重新计算搜索匹配、章节关系和活动样式依赖。

相关位置：

- `src/components/TranscriptDetail.tsx`
- `src/components/AudioPlayer.tsx`

现有 `content-visibility` 只能减少浏览器绘制，不能阻止 React 为全部节点执行协调。

## 3.2 P0：每次编辑都会复制并保存完整结果

编辑一个段落时，当前链路会：

1. 复制完整 `segments` 数组。
2. 重新拼接完整 `result.text`。
3. 更新 `selectedResult`。
4. 更新队列中的结果副本。
5. 更新历史数组中的结果副本。
6. 450ms 后序列化并重写完整 `history.json`。

历史数据增长后，输入延迟、内存分配、垃圾回收和磁盘写入都会放大。

相关位置：

- `src/components/TranscriptDetail.tsx`
- `src/App.tsx`
- `electron/main.ts`

## 3.3 P0：等待 RPM 的 ASR 请求提前持有大块 Base64 数据

每个音频切片会在进入 RPM 等待前读取完整文件并转成 Base64。

以 6.4 MiB 切片估算：

- 原始 Buffer 约 6.4 MiB。
- Base64 字符串约 8.5 MiB。
- JSON 请求体可能再生成一个大字符串副本。

延迟感知算法把在途并发提高到 20–30 时，等待中的请求可能同时保留数百 MiB 数据。

相关位置：`electron/main.ts` 的 `requestTranscript()`。

## 3.4 P1：组件 `memo` 被不稳定函数属性抵消

多数子组件已经包裹 `memo`，但 `App.tsx` 在 JSX 中持续创建新的内联回调。父级任何状态变化都会让这些属性引用变化，导致子组件无法跳过渲染。

当前进度事件还共用一个 `requestAnimationFrame`；新事件会取消同一帧中较早的事件。多个文件同时报告进度时可能丢失更新。这是性能优化前必须修复的正确性问题。

## 3.5 P1：启动阶段向 Renderer 传输完整历史正文

启动请求已经并行，但 Renderer 仍一次接收：

- 设置。
- 全部历史正文、段落和切片。
- AI Provider。
- 完整媒体库索引。

当前历史只有 12 条就达到 3.2 MB。增长到数百条后，会增加：

- 主进程 JSON 解析。
- Electron IPC 结构化克隆。
- Renderer 解析和常驻内存。
- React 列表派生计算。

## 3.6 P1：媒体库搜索和列表不能线性扩展

当前存在以下问题：

- 文件名搜索会同时扫描完整历史正文。
- 每个文件夹都重新遍历全部资产计算数量。
- 选中状态使用数组 `includes`，大量选择时为线性查找。
- 全部媒体行仍会生成 React DOM。
- 批量导入通过逐文件 `find` 检测重复，规模增大后接近 O(n²)。
- 媒体索引在多个 IPC 操作中重复读取磁盘。

相关位置：

- `src/components/MediaLibraryView.tsx`
- `electron/media-library.ts`
- `electron/main.ts`

## 3.7 P2：AI 流式 Markdown 重复解析

AI 增量已经按动画帧批量刷新，但每次内容增长仍会重新解析完整 Markdown。长表格、任务列表和代码块会放大成本。

相关位置：

- `src/components/AIChatPanel.tsx`
- `src/components/MarkdownMessage.tsx`

## 3.8 P2：启动包没有功能级拆分

以下功能被静态导入到主包：

- AI 对话。
- `react-markdown`、GFM、HTML 清洗。
- 设置窗口。
- 媒体库。
- 智能速览相关界面。

即使用户不打开对应功能，Renderer 启动时也会解析这些模块。

## 3.9 P2：Liquid Glass 存在重复合成成本

现有 Liquid Glass 设计需要保留，但多层面板、列表和固定播放器同时使用模糊与阴影，会增加滚动时的 GPU 合成压力。

优化目标不是移除玻璃风格，而是让真实模糊只出现在最有价值的顶层表面，子卡片使用低成本的透明渐变、描边和内高光近似。

## 4. 总体实施原则

1. 先测量，后修改。
2. 每项优化都必须有可重复的性能夹具或自动化回归测试。
3. 先修正确性风险，再做速度优化。
4. 每个阶段独立分支、独立 PR、独立回退点。
5. 不同时重构渲染、数据格式和 ASR 请求链。
6. 高风险数据迁移必须支持中断恢复和旧版本回退。
7. 性能优化不得降低识别质量、转写完整性或用户数据安全。
8. 不以“减少功能”作为性能优化手段。

## 6. 阶段 1：超长转写渲染与编辑

建议分支：`codex/perf-transcript-rendering`

## 6.1 隔离高频播放状态

- `AudioPlayer` 自己维护高频 `currentTime`。
- 父组件只在跨越段落边界时接收新的 `activeSegmentId`。
- 校正按钮通过播放器控制器/ref 读取当前时间。
- 不再把每次 `timeupdate` 写入 `TranscriptDetail` 顶层状态。
- 对段落起止时间建立有序索引并使用二分查找，将活动段定位从 O(n) 降为 O(log n)。
- 当前段高亮只更新旧段和新段。

## 6.2 虚拟化时间轴

先制作小型原型比较动态高度方案：

- 优先选用支持动态测量和 `scrollToIndex` 的轻量虚拟列表。
- 只挂载可见区域和少量 overscan。
- 缓存章节卡、失败片段和 textarea 的实际高度。
- 搜索跳转、章节跳转、播放跟随统一使用段落索引定位。
- 用户正在编辑的行不得因轻微滚动被卸载。
- 字体、AI 面板宽度、窗口尺寸变化后正确刷新高度缓存。

若动态 textarea 虚拟化无法达到稳定验收标准，则采用低风险过渡方案：分段窗口渲染加 `content-visibility`，但不自研复杂虚拟列表。

## 6.3 段落级编辑

- 每个段落组件维护本地 draft。
- 输入过程中只重渲染当前段落。
- 600ms 空闲、失焦、切换记录、关闭详情和退出应用时强制提交。
- IPC 改为 `transcriptId + segmentId + patch`。
- 不再为单段输入发送完整 `TranscriptResult`。
- `result.text` 在保存检查点、导出、复制和 AI 请求时按需生成。
- 正文修改后只把旧智能速览标为过期，不在每次按键时重算。

## 6.4 稳定组件边界

- 将传给 `memo` 组件的业务回调改为稳定 `useCallback`。
- 抽离 `TranscriptSegmentRow`、`SearchResultList`、`AnalysisPanel`。
- 避免 JSX 中创建会破坏 memo 的函数和对象属性。
- 只对有实测收益的组件使用 memo，避免无意义的全面 memo 化。

## 6.5 修复进度事件批处理

现有单一 RAF 会用新事件覆盖同帧旧事件。改为：

- 使用 `Map<fileId, ProgressEvent>` 收集一帧内最新状态。
- 下一帧一次性合并所有文件更新。
- 不同文件的进度互不覆盖。
- 组件卸载时清理 RAF 和缓存 Map。

## 7. 阶段 2：ASR 内存与主进程吞吐

建议分支：`codex/perf-asr-memory`

## 7.1 延后大对象创建

请求顺序调整为：

```text
等待 RPM 与全局冷却许可
→ 获取本地内存额度
→ 读取音频文件
→ 转换 Base64
→ 创建 JSON 请求体
→ 发送请求
→ 响应完成后立即释放大对象和内存额度
```

等待 RPM 的任务不再提前持有音频 Buffer、Base64 和 JSON 请求体。

## 7.2 字节级内存闸门

该闸门只限制本地编码内存，不是 MiMo API 的固定并发上限：

- 按待编码音频字节数申请额度。
- 初始总预算建议 128–192 MiB。
- 根据系统可用内存动态调整。
- 没有额度的任务延后编码，但仍保留调度顺序。
- 429/RPM 等待继续不消耗真实错误重试次数。
- 任务取消时立即释放额度。

## 7.3 流式 Base64 兼容性原型

单独验证是否可通过流式请求体生成 JSON/Base64：

- 检查小米按量与 Token Plan 两个入口。
- 测量峰值内存、CPU 和失败率。
- 验证代理、重试和取消兼容性。
- 只有在真实接口稳定通过时才启用。
- 若服务端兼容性不足，保留“RPM 后编码 + 字节预算”方案。

## 7.4 日志批量写入

- 日志先进入内存队列。
- 每 100–250ms 或达到一定行数时批量 append。
- 缓存当前日志大小，不为每一行调用 `stat`。
- 轮转时再执行文件检查。
- 应用退出和异常结束路径强制 flush。
- 继续禁止记录 API Key、Base64、转写正文和完整路径。

## 8. 阶段 3：历史存储与启动加载

建议分支：`codex/perf-history-store`

## 8.1 启动仅加载摘要

新增轻量数据结构：

```ts
interface TranscriptSummary {
  id: string
  fileName: string
  createdAt: string
  duration: number
  outcome?: 'complete' | 'partial' | 'failed'
  segmentCount: number
  mediaId?: string
  analysisStatus?: 'none' | 'ready' | 'stale'
}
```

启动阶段仅加载：

- 应用设置。
- 历史摘要。
- 媒体库摘要。

完整转写在打开记录时按 ID 加载。

AI Provider 设置在用户打开 AI 对话或 AI 设置时加载；加载失败不影响上传、历史和播放界面。

## 8.2 历史记录分离存储

从单一 `history.json` 迁移为：

```text
history/
  index.json
  records/
    <transcript-id>.json
```

收益：

- 修改一条记录只写一个文件。
- 删除记录不再序列化全部历史。
- Main 无需长期缓存全部正文。
- Renderer 只保留当前详情和媒体摘要。
- 历史规模不会线性放大每次编辑写盘成本。

## 8.3 数据安全和旧版本回退

- 迁移前复制原 `history.json`。
- 原文件至少保留一个正式版本周期。
- 新目录先写入临时位置。
- 校验记录数、ID、总文本长度和关键字段后再原子切换。
- 迁移中断或校验失败时继续使用旧格式。
- 新版本提供一次性兼容导出，可供旧安装包恢复。
- 不允许解析失败后以空数组覆盖历史。
- 媒体文件索引迁移遵循同样的备份和原子切换原则。

## 9. 阶段 4：媒体库扩展性

建议分支：`codex/perf-media-library`

- Renderer 中的媒体库只使用转写摘要，不携带完整正文。
- 文件夹数量通过一次遍历生成 `Map<folderId, count>`。
- 资产按 ID 建立 Map。
- 选择状态使用 Set。
- 媒体行采用虚拟滚动。
- 搜索结果设置可视上限并提供继续加载。
- 默认文件名搜索不扫描完整转写正文。
- 全文搜索作为明确独立功能，通过主进程索引或按需加载完成。
- 主进程为媒体索引增加写穿缓存。
- 媒体库位置变化时显式失效缓存。
- 批量导入建立来源签名 Map，消除逐文件 `find`。
- 文件复制保持顺序或有限并行，避免同时复制大量大文件。
- FFprobe 并发根据 CPU 和 ASR 是否运行动态控制。
- 导入、复制、探测分别报告进度，避免用户误判为界面卡死。

## 10. 阶段 5：AI、搜索、启动包与 Liquid Glass

建议分支：`codex/perf-bundle-and-glass`

## 10.1 AI 流式渲染

- 流式生成期间使用安全纯文本或轻量预览。
- 完成后才解析 Markdown。
- `remark`、`rehype` 和 Markdown 组件配置提升为模块级常量。
- Markdown 渲染器动态加载。
- 用户不在对话底部时停止自动滚动。
- 长会话只挂载最近消息，旧消息按需展开。
- 转写上下文字符串可按 `transcriptId + revision` 缓存，正文未变化时复用。

## 10.2 搜索优化

- 打开转写时建立按段落缓存的小写搜索文本。
- 只在编辑对应段落后失效该段索引。
- 搜索结果列表虚拟化或限制首批显示数量。
- 保留结果总数、上一项/下一项和章节分组。
- 极长记录达到阈值后，评估 Web Worker 搜索；当前规模若主线程已满足指标则不引入 Worker。

## 10.3 功能级动态加载

按需加载：

- AI 对话与 Markdown。
- 设置窗口。
- 媒体库。
- 智能速览的低频管理功能。

使用稳定的加载占位，避免打开功能时界面跳动。

## 10.4 字体资源

- 将通用 `@fontsource-variable/inter` 改为实际使用的字符子集。
- 或优先使用 Windows 系统字体，Inter 只保留必要拉丁字符。
- 不加载未使用的希腊、越南、斯拉夫扩展字库。
- 验证中英文混排、数字、代码和 Markdown 字体回退。

## 10.5 Liquid Glass 合成优化

保持现有 Apple Liquid Glass 视觉，不添加鼠标跟随或 WebGL：

- 真实 `backdrop-filter` 仅用于导航、主面板、弹窗和固定播放器等顶层表面。
- 子卡片改用透明渐变、边缘高光和低成本阴影。
- 滚动列表行不建立独立模糊层。
- 大型滚动区域增加经过测试的 layout/paint containment。
- 面板拖动期间临时降低模糊和阴影成本，结束后恢复。
- 保留减少透明度、减少动画和高对比度降级。
- 在浅色、深色和不同玻璃强度下分别测试 GPU 合成。

## 11. 初始验收指标

阶段 0 实测后可以校准绝对阈值，但优化后不得低于以下目标：

| 场景 | 目标 |
|---|---|
| 1,200 段详情页播放 | React commit p95 小于 16ms |
| 播放进度更新 | 不触发详情页整体重渲染 |
| 60,000 字记录输入延迟 | input-to-paint p95 小于 50ms |
| 单段编辑 | 只重渲染当前行及必要状态组件 |
| 10,000 条媒体列表滚动 | 目标 55–60 FPS |
| 转写搜索首批结果 | 小于 100ms |
| 冷启动 | 相比阶段 0 基线至少降低 30% |
| Renderer 初始历史传输 | 从完整正文降为摘要，目标小于 300 KB |
| 等待 RPM 的 ASR 请求 | 不持有 Base64 请求体 |
| 高并发 ASR 额外内存 | 严格受字节预算控制 |
| 历史更新 | 不再重写完整历史库 |
| AI 流式输出 | 生成期间不重复解析完整 Markdown |
| 正确性回归 | 当前 86 项测试保持通过 |
| 新增性能测试 | 固定夹具可在 CI 中重复执行 |

## 14. 发布与回退

## 14.1 发布流程

1. 所有阶段合并后创建 `v0.12.0-rc.1` 注释标签。
2. GitHub Actions 构建 Windows 安装包。
3. 使用真实长转写、批量媒体和长时间播放进行 RC 验收。
4. 发现问题时只回退对应 PR，并发布新的 RC。
5. 验收通过后创建 `v0.12.0` 注释标签。
6. 创建 GitHub Release。

Release 应包含：

- Windows 安装包。
- SHA-256。
- 性能前后对比报告。
- 历史数据迁移说明。
- 回退说明。
- 已知限制。

## 14.2 回退层级

### UI 或渲染错误

- Revert 对应渲染 PR。
- 数据格式未变时无需恢复用户数据。

### ASR 内存或请求错误

- 通过内部兼容开关恢复旧请求路径。
- 或 revert `perf-asr-memory` PR。
- 识别结果和历史数据不受影响。

### 历史存储迁移错误

- 停止新格式写入。
- 恢复迁移前备份。
- 安装上一正式版本。
- 使用兼容导出恢复新版本期间的新增记录。

### 整体重大错误

- 从上一正式 Git tag 重新构建安装包。
- 恢复对应版本的数据备份。
- GitHub Release 保留旧安装包和 SHA，不覆盖同名产物。
